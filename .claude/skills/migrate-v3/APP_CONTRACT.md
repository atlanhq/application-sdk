# App Contract Skill

Set up or migrate an Atlan native app to use the `app-contract-toolkit` as a remote PKL package. One `contract/app.pkl` generates all configuration artifacts, and the SDK auto-serves them.

## Target Project Structure

After running this skill, the app repo should look like this:

```
your-app/
├── contract/
│   ├── PklProject                          # Package dependency declaration
│   ├── PklProject.deps.json                # Resolved lock file (generated by pkl project resolve)
│   ├── app.pkl                             # App contract definition (YOU WRITE THIS)
│   └── generated/                          # Generated artifacts (DO NOT EDIT)
│       ├── {name}.json                     # Workflow config (setup form)
│       ├── atlan-connectors-{name}.json    # Credential config (auth form)
│       ├── manifest.json                   # AE DAG (extract → publish)
│       └── input.py                        # Python typed Input class
├── app/
│   ├── contracts/
│   │   └── _input.py                       # Generated typed dataclass (moved from contract/generated/)
│   ├── handlers/                           # No get_configmap/get_manifest overrides needed
│   └── workflows/                          # Contains the workflow class referenced by workflowType
├── frontend/
│   └── static/                             # App playground (npx @atlanhq/app-playground install-to)
├── pyproject.toml                          # SDK + poe generate task
├── uv.lock
└── ...
```

### Files You Create/Edit

| File | Action | Description |
|---|---|---|
| `contract/PklProject` | Create | Declares toolkit package dependency |
| `contract/app.pkl` | Create or migrate | App contract — metadata, credentials, workflow form |
| `pyproject.toml` | Edit | Add `poe generate` task |

### Files That Get Generated

| File | Generated By | Served At | Description |
|---|---|---|---|
| `contract/generated/{name}.json` | `pkl eval` | `GET /workflows/v1/configmap/{name}` | Setup form schema (steps, properties, anyOf) |
| `contract/generated/atlan-connectors-{name}.json` | `pkl eval` | `GET /workflows/v1/configmap/atlan-connectors-{name}` | Credential form (auth types, nested fields) |
| `contract/generated/manifest.json` | `pkl eval` | `GET /manifest` | AE DAG template with `{{param}}` placeholders |
| `contract/PklProject.deps.json` | `pkl project resolve` | N/A | Lock file with resolved checksums |
| `app/contracts/_input.py` | `poe generate` (moved) | N/A | Typed `@dataclass` for workflow args |

### Files To Remove (If Migrating)

| File/Pattern | Why |
|---|---|
| `app/templates/` | Replaced by `contract/generated/` |
| `get_configmap()` overrides in handlers | SDK auto-serves from `contract/generated/` |
| `get_manifest()` overrides in handlers | SDK reads `contract/generated/manifest.json` |

## Workflow

### 0. Discover Existing State

Before making any changes, check what already exists in the app repo:

```bash
# Check for existing contract directory
ls contract/ 2>/dev/null

# Check if app.pkl exists and what import style it uses
head -5 contract/app.pkl 2>/dev/null

# Check for old config serving patterns (handler overrides, templates)
grep -rn "get_configmap\|get_manifest\|templates/" app/handlers/*.py 2>/dev/null

# Check current SDK version in pyproject.toml
grep "atlan-application-sdk" pyproject.toml

# Check for existing generated files
ls contract/generated/ 2>/dev/null

# Check if PklProject exists
cat contract/PklProject 2>/dev/null

# Check for existing poe generate task
grep "generate" pyproject.toml

# Check for existing template files — these are the REFERENCE for what the generated
# contract output must match. Save them for comparison BEFORE deleting.
ls app/templates/ 2>/dev/null
ls frontend/static/ 2>/dev/null
# If templates exist, save their JSON content for later diff against contract output
for f in app/templates/*.json 2>/dev/null; do
    echo "=== OLD TEMPLATE: $(basename $f) ==="
    python3 -c "import json; print(json.dumps(json.load(open('$f')), indent=2))" 2>/dev/null | head -30
done

# CRITICAL: Verify SDK version is 2.8.0 or later (contract auto-serving requires it)
# The SDK must be >= 2.8.0 for:
#   - Auto-serving configs from contract/generated/ (no handler overrides needed)
#   - Proper manifest.json loading
#   - SageV2 preflight check routing
SDK_VERSION=$(grep "atlan-application-sdk" pyproject.toml | grep -oE '[0-9]+\.[0-9]+\.[0-9]+')
LATEST_SDK=$(gh release list --repo atlanhq/application-sdk --limit 1 --json tagName --jq '.[0].tagName' 2>/dev/null | tr -d 'v')
echo "Current SDK: $SDK_VERSION"
echo "Latest SDK: $LATEST_SDK"
if [ "$(printf '%s\n' "2.8.0" "$SDK_VERSION" | sort -V | head -1)" != "2.8.0" ]; then
    echo "BLOCKER: SDK $SDK_VERSION is older than 2.8.0 — contract auto-serving won't work"
    echo "MUST update pyproject.toml to use SDK >= 2.8.0 (latest: $LATEST_SDK)"
    echo "DO NOT PROCEED until SDK is updated"
fi

# IMPORTANT: Check legacy Argo configmap for sageTemplate (preflight checks)
# This is stored separately from the form properties and is easy to miss during migration
PARENT_DIR=$(dirname "$(git rev-parse --show-toplevel)")
CONNECTOR_NAME=$(basename "$(git rev-parse --show-toplevel)" | sed 's/atlan-\(.*\)-app/\1/')
grep -l "sageTemplate" "$PARENT_DIR/marketplace-packages/packages/atlan/connectors/configmaps/"*.yaml 2>/dev/null | xargs grep -A 5 "sageTemplate" 2>/dev/null | head -20
```

Based on what exists, determine the migration path:
- **No contract/ dir** → full setup from scratch (Steps 1–7)
- **Has app.pkl with local paths** (`../../app-contract-toolkit/src/...`) → migrate imports + add PklProject (Steps 1–7)
- **Has app.pkl with package imports** (`@app-contract-toolkit/...`) → verify and regenerate (Steps 4–7)

### 1. Create `contract/PklProject`

**Always use the latest toolkit release.** Fetch the latest version:

```bash
gh release list --repo atlanhq/app-contract-toolkit --limit 1 --json tagName --jq '.[0].tagName'
```

Then create the PklProject with that version (strip the `v` prefix):

```pkl
amends "pkl:Project"

dependencies {
  ["app-contract-toolkit"] {
    uri = "package://atlanhq.github.io/app-contract-toolkit/app-contract-toolkit@<VERSION>"
  }
}
```

Then resolve the dependency:

```bash
cd contract && pkl project resolve
```

This generates `PklProject.deps.json` (the lock file).

**To bump an existing project to a newer toolkit version**, update the version in `PklProject`, then re-resolve:
```bash
cd contract && pkl project resolve && pkl eval -m generated app.pkl
```

**Before bumping or creating a contract, always read the toolkit reference docs** to understand
available widgets, field types, and any recent changes. The toolkit evolves — new widget types
get added, field rendering may change, and new PKL classes may be available that simplify your
contract.

```bash
# Read the reference docs to understand all available Config types, FieldSpec options, and widget mappings
gh api repos/atlanhq/app-contract-toolkit/contents/docs/reference.md -H "Accept: application/vnd.github.raw" | head -500

# Read the README for high-level usage patterns and examples (do this when setting up from scratch)
gh api repos/atlanhq/app-contract-toolkit/contents/README.md -H "Accept: application/vnd.github.raw" | head -300

# Check release notes for the version you're bumping to — look for new features, breaking changes, fixes
gh release view <version> --repo atlanhq/app-contract-toolkit
```

If the toolkit reference shows new Config types or FieldSpec options that weren't available when
the contract was originally written, consider whether the app.pkl should be updated to use them
(e.g., a new widget type that better fits an existing field, or a fix for a rendering issue).

### 2. Create or Migrate `contract/app.pkl`

#### **CHECKPOINT: Ask about publish step (CSA packages only)**

If the app being migrated or built is a **CSA package** (i.e. a Python script migrated from `marketplace-csa-scripts`), ask the user:

> "Does this workflow need to publish metadata to Atlas as a separate AE step, or does the entire workflow run as its own standalone activity (no publish node needed)?"

**Context for the user:** Legacy CSA Python scripts often included a `publish_to_atlas()` call inline. When migrating to the App SDK, the AE orchestrator adds a separate `publish` node in `manifest.json` that runs after `extract`. However, for CSA packages that run as a self-contained workflow (e.g. they write metadata directly via the SDK or pyatlan, not via JSONL → Publish App), the publish node is **not needed and must be removed**.

- If the user says **publish step is NOT required** → set `hasPublishStep = false` in `app.pkl`. The generated `manifest.json` will only contain the `extract` node and no publish-related fields will appear in `input.py`.
- If the user says **publish step IS required** → leave `hasPublishStep` at its default (true) and proceed as normal.

**If creating from scratch**, gather information from:
- Existing handler code (`app/handlers/`) for auth types, credential fields
- Existing configmap templates (`app/templates/`) or legacy YAML configs
- Workflow class name from `app/workflows/`
- Connector type from the app name

**If migrating existing app.pkl**, update the imports:

```pkl
// Replace these local imports:
amends "../../app-contract-toolkit/src/NativeApp.pkl"
import "../../app-contract-toolkit/src/Config.pkl"
import "../../app-contract-toolkit/src/Connectors.pkl"

// With package imports:
amends "@app-contract-toolkit/NativeApp.pkl"
import "@app-contract-toolkit/Config.pkl"
import "@app-contract-toolkit/Connectors.pkl"
```

#### Critical Naming Rules

**`name`** — The app slug (e.g., `"redshift"`, `"matillion"`). This drives everything:
- Workflow configmap file: `{name}.json`
- Credential configmap file: `atlan-connectors-{name}.json`
- Task queue: `atlan-{name}-{deployment_name}`
- Service URL: `http://{name}.{name}-app.svc.cluster.local`

**`credType` in `Config.CredentialInput`** — Must be `"atlan-connectors-{name}"` to match the generated credential config file. This is NOT auto-derived — you must set it explicitly:
```pkl
["credential-guid"] = new Config.CredentialInput {
  title = ""
  credType = "atlan-connectors-<name>"  // MUST match the name field
}
```

**`credentialType` in `Config.APITree` and `Config.SqlTree`** — Same rule, must be `"atlan-connectors-{name}"`:
```pkl
["include-filter"] = new Config.SqlTree {
  credentialType = "atlan-connectors-<name>"  // MUST match
  excludePatterns = List("information_*")     // Schema-level exclude patterns
  databaseExcludePatterns = List("jmx", "memory", "system")  // Database/catalog-level exclude patterns (multi-catalog connectors)
}
```

#### Critical: Conditional Fields and Default Values

When using `anyOf` rules to conditionally show fields (e.g., show credential fields only when extraction-method is "direct"), the controlling field **must** have its `default` at the property level. The frontend reads `default` from the property level to evaluate `anyOf` conditions.

**Use `Config.Radio` (not `Config.TextInput`) for fields that control `anyOf` rules:**

```pkl
// CORRECT — Radio puts default at property level
["extraction-method"] = new Config.Radio {
  title = "Extraction method"
  possibleValues { ["direct"] = "Direct"; ["s3"] = "Offline" }
  default = "direct"
}

// WRONG — TextInput puts default inside ui (won't trigger anyOf)
["extraction-method"] = new Config.TextInput {
  title = "Extraction method"
  defaultValue = "direct"
  hide = true
}
```

If you only support one extraction method and want it hidden, you can still use `TextInput` but then do NOT use `anyOf` rules that depend on its value — instead make `credential-guid` always visible without conditions.

#### Critical: Step Descriptions Must Be Included

Every task in `uiConfig.tasks` **must** have a `description` field. This text appears below the step title in the left sidebar of the setup wizard. Without it, the sidebar shows bare titles with no context, making the UI feel incomplete compared to Argo-based apps.

```pkl
uiConfig {
  tasks {
    ["Credential"] {
      description = "Credential Details"
      inputs { ... }
    }
    ["Connection"] {
      description = "Connection Details"
      inputs { ... }
    }
    ["Metadata"] {
      description = "Metadata"
      inputs { ... }
    }
  }
}
```

The `description` field defaults to `""` in `Config.UIStep`, so omitting it produces an empty subtitle. Always set it explicitly for every step.

#### Critical: Preflight Checks (SageV2) Must Be Included

Every connector **must** have a `preflight-check` widget in the Metadata step. This renders the
"Preflight checks" panel in the setup wizard that validates permissions before the workflow runs.
Without it, users have no way to verify connectivity/permissions before starting extraction.

**`preflight-check` must be the LAST input in the Metadata step.** The UI renders fields in the
order they appear in the `inputs` map. Preflight checks should appear after all filter and
config fields (include-filter, exclude-filter, advanced-config, etc.) — never in the middle.

**Always check the legacy Argo configmap for `sageTemplate`** to find the check names. The
`sageTemplate` JSON keys map directly to `Config.SageV2` check names:

```yaml
# Legacy Argo sageTemplate (in marketplace-packages configmap YAML)
sageTemplate: |
  {
    "catalogsCheck": { ... },
    "schemasCheck": { ... }
  }
```

Maps to:
```pkl
// In app.pkl Metadata task
["preflight-check"] = new Config.SageV2 {
  title = ""
  checks {
    new Dynamic { id = 1; name = "catalogsCheck"; title = "Catalogs and schemas"; loaderText = "Checking permissions" }
  }
}
```

**For SQL connectors**, the check name is typically `catalogsCheck` or `schemasCheck`.
**For REST connectors**, the check name is typically `loginCheck` or `connectivityCheck`.

The `name` field in each check **must match** what the SDK's `/workflows/v1/check` endpoint
handler returns — the SDK uses this name to map the check result to the UI indicator.

#### Critical: Workflow Must Return Outputs

Your workflow's `run()` method **must return a dict** with outputs for the publish node. If `run()` returns `None`, the AE publish step fails with `ValueError: Jsonpath '$.extract.outputs.connection_qualified_name' did not match any value`.

For SDK base workflows, ensure you return the result of `run_exit_activities`:
```python
# CORRECT
return await self.run_exit_activities(workflow_args)

# WRONG — discards the output dict
await self.run_exit_activities(workflow_args)
```

#### Utility Apps: Disabling the Publish Step

By default, every app gets an `extract → publish` DAG pipeline. Utility apps (duplicate detector, bulk updater) that don't extract metadata for publishing to Atlas should disable it:

```pkl
hasPublishStep = false       // No publish node in manifest, no publish fields in input.py
hasCredentialConfig = false   // No credential config generated (if the app doesn't use credentials)
```

When `hasPublishStep = false`:
- The manifest DAG only contains the `extract` node
- Publish-related fields (`output_dir`, `checkpoint_dir`, `load_to_atlan`, `publish_dry_run`) are omitted from the generated Input class
- The workflow's `run()` does NOT need to return publish outputs

#### Critical: `extraction-method` Field Required for Sage/Preflight Routing

Every native app **must** include an `extraction-method` field in the **Credential step**, even
if the connector only supports direct extraction. Heracles uses this field to decide whether to
route sage/preflight requests to the native app's `/workflows/v1/check` endpoint or fall through
to the legacy `sageTemplate` curl execution.

Without this field, `extractionMethod` is empty in Heracles's sage handler, which fails the
routing condition:

```go
// heracles/handler/credential.go — sage routing decision
if appServiceURL != "" && (extractionMethod == "direct" || extractionMethod == "query_history") {
    // → route to native app /workflows/v1/check
}
// else → legacy sageTemplate (fails with "unexpected end of JSON input" for native apps)
```

**For connectors with only one extraction method** (most REST connectors):
```pkl
["extraction-method"] = new Config.TextInput {
  title = "Extraction method"
  defaultValue = "direct"
  hide = true
}
```

**For connectors with multiple extraction methods** (e.g., direct + S3 offline):
```pkl
["extraction-method"] = new Config.Radio {
  title = "Extraction method"
  possibleValues { ["direct"] = "Direct"; ["s3"] = "Offline" }
  default = "direct"
  required = true
}
```

**Symptom if missing:** Preflight checks fail with `"unexpected end of JSON input"` from
`handler/credential.go`. The `POST /workflows/v1/check` request never reaches the app pod.
Configmap, auth, and metadata endpoints all work fine — only sage is affected.

### 3. Update `pyproject.toml`

**Exclude `_input.py` from linters and type checkers.** The generated `_input.py` uses
experimental SDK imports and hyphenated field names that will fail ruff and pyright. Add these
exclusions to `pyproject.toml`:

```toml
[tool.ruff]
exclude = ["app/contracts/_input.py"]

[tool.pyright]
exclude = [
    ".github/**",
    "app/contracts/_input.py",
]
```

If `[tool.pyright]` already has an `exclude` list, just append `"app/contracts/_input.py"` to it.
If `[tool.ruff]` already exists, append to its `exclude` list.

**Add the generate task.** Try the standard command first:

```toml
[tool.poe.tasks]
# Contract generation
generate.shell = "cd contract && pkl eval -m generated app.pkl && mkdir -p ../app/contracts && mv generated/input.py ../app/contracts/_input.py"
```

Then test it:

```bash
uv run poe generate
```

If it fails with an **SSL handshake error** (common on corporate VPNs), switch to the CA-certificate variant:

```toml
[tool.poe.tasks]
# Contract generation (with CA cert for VPN/corporate proxy environments)
generate.shell = "cd contract && pkl eval --ca-certificates ~/.pkl-ca-bundle.pem -m generated app.pkl && mkdir -p ../app/contracts && mv generated/input.py ../app/contracts/_input.py"
```

To create the CA bundle if it doesn't exist:
```bash
security find-certificate -a -p /System/Library/Keychains/SystemRootCertificates.keychain > ~/.pkl-ca-bundle.pem
security find-certificate -a -p ~/Library/Keychains/login.keychain-db >> ~/.pkl-ca-bundle.pem 2>/dev/null
```

### 4. Generate Artifacts

```bash
uv sync
uv run poe generate
```

This produces 4 artifacts:

| File | Location | Purpose |
|---|---|---|
| `{name}.json` | `contract/generated/` | Workflow config — setup form with steps, properties, anyOf |
| `atlan-connectors-{name}.json` | `contract/generated/` | Credential config — auth types, fields |
| `manifest.json` | `contract/generated/` | AE DAG — extract → publish pipeline |
| `_input.py` | `app/contracts/` | Typed Python `@dataclass` for workflow args |

### 5. Validate Contract Output Against Old Templates, Then Remove (If Migrating)

If the app previously served configs via handler overrides or template files, **validate the
generated contract output matches the old templates before deleting them**.

```bash
# Check what old patterns exist
grep -rn "get_configmap\|get_manifest\|templates/" app/handlers/*.py 2>/dev/null
ls app/templates/ 2>/dev/null
```

**Compare old vs new before deleting:**
```bash
# For each old template JSON, diff against the new contract/generated/ equivalent
```

## Common Pitfalls

### 1. Credential widget doesn't render
**Cause:** `anyOf` rule depends on a field whose `default` is inside `ui` instead of property level.
**Fix:** Use `Config.Radio` for fields that control `anyOf` conditions, not `Config.TextInput`.

### 2. CA certificate / textarea field renders empty
**Cause:** The generated JSON has `"widget": "textBox"` (toolkit < v0.2.4) but the frontend expects `"TextInput"`.
**Fix:** Upgrade toolkit to v0.2.4+.

### 3. `credentialType` mismatch — credential form doesn't load
**Cause:** `credType` in `CredentialInput` or `credentialType` in `APITree`/`SqlTree` doesn't match `"atlan-connectors-{name}"`.
**Fix:** Ensure all credential type references use `"atlan-connectors-{name}"` where `{name}` matches your `name` field.

### 4. Publish step fails with `Jsonpath did not match any value`
**Cause:** Either the workflow's `run()` method doesn't return the output dict, or the app is a utility app that shouldn't have a publish step at all.
**Fix:** For connector apps, return the result of `run_exit_activities()`:
```python
return await self.run_exit_activities(workflow_args)
```
For utility apps that don't publish metadata, set `hasPublishStep = false` in `app.pkl` to suppress the publish node entirely.

### 5. Old generated files conflict after rename
**Cause:** Changed `name` in app.pkl but old `{old-name}.json` files still exist in `contract/generated/`.
**Fix:** Delete old files after renaming.

### 6. Preflight checks panel missing from setup wizard
**Cause:** The `preflight-check` SageV2 widget was not added to the Metadata task in `app.pkl`. This is easy to miss because the legacy Argo config defines it as a separate `sageTemplate` field in the configmap YAML, not inline with the form properties.
**Fix:** Always check the legacy Argo configmap for a `sageTemplate` field and port each check key to a `Config.SageV2` entry. See "Critical: Preflight Checks" section above.

### 7. Sage/preflight fails with "unexpected end of JSON input" — Heracles can't resolve app
**Cause:** Heracles can't resolve the credential configmap name to the native app. Falls through to legacy path which needs `sageTemplate` (native apps don't have one).
**Fix:** Ensure Heracles has the `deriveAppName` fallback fix (BLDX-776). Verify with pod logs that `POST /workflows/v1/check` reaches the app server.

### 8. Preflight fails with "unexpected end of JSON input" — `extraction-method` missing
**Cause:** No `extraction-method` field in the Credential step. Heracles uses this field to route sage requests to native apps. Without it, `extractionMethod` is empty and Heracles falls through to the legacy `sageTemplate` which doesn't exist for native apps.
**Fix:** Add a hidden `extraction-method` field with `defaultValue = "direct"` to the Credential task. See "Critical: `extraction-method` Field Required" section above.

## Post-Migration Verification Checklist

**Before deploying, walk through every item below.** Each comes from a real failure — the app
appeared to work locally but failed on a live tenant or during AE orchestration.

### V1. Workflow Must Return an Output Dict for AE Publish

**What fails:** AE's publish node resolves `$.extract.outputs.connection_qualified_name`,
`$.extract.outputs.transformed_data_prefix`, etc. via jsonpath from the extract workflow's
return value. If the workflow returns `None`, all jsonpath expressions match nothing and AE
raises `ApplicationError("Workflow failed due to activity failure(s): ['execute_workflow']")`.

**Why it's easy to miss:** The workflow runs successfully — all activities complete, metadata
is extracted and transformed. The failure only surfaces when AE tries to start the publish
node, which is a separate Temporal workflow managed by AE, not the connector app.

**How to verify:**
- Check the workflow's `run()` method return type — it must be `Optional[Dict[str, Any]]`, not `None`.
- The return dict must include exactly these keys (AE resolves them via jsonpath from `manifest.json`):
```python
return {
    "transformed_data_prefix": "...",
    "connection_qualified_name": "...",
    "publish_state_prefix": "persistent-artifacts/apps/atlan-publish-app/state/{connection_qn}/publish-state",
    "current_state_prefix": "argo-artifacts/{connection_qn}/current-state",
}
```
- Cross-reference with `contract/generated/manifest.json` — every `$.extract.outputs.<key>` in the publish node's inputs must exist in the return dict.
- If the connector extends a base SDK workflow (like `SQLMetadataExtractionWorkflow`), the base class handles this. Custom workflows must build it manually.

### V2. Relationship References Must Use `ref_by_qualified_name()`

**What fails:** The publish app sends entities to Atlas. Atlas expects relationship references
to have `uniqueAttributes.qualifiedName`. If references use `attributes.qualifiedName` instead,
Atlas returns `ObjectId is not valid` and every entity with that relationship fails to publish.

**Why it's easy to miss:** `to_atlas_dict()` serializes successfully, the JSONL looks valid,
the entity has a qualifiedName — but it's in the wrong JSON key. The failure only appears in
the publish app logs, not the connector app.

**Wrong pattern:**
```python
ref = ModeCollection()
ref.qualified_name = "some/qn"  # → {"attributes": {"qualifiedName": "some/qn"}}
asset.mode_collections = [ref]
```

**Correct pattern:**
```python
asset.mode_collections = [
    ModeCollection.ref_by_qualified_name("some/qn")  # → {"uniqueAttributes": {"qualifiedName": "some/qn"}}
]
```

**How to verify:**
- Search the transformer code for any `ref = SomeAsset()` followed by `ref.qualified_name = ...`. Every one is a bug.
- After a local workflow run, inspect the transformed JSONL:
```bash
head -1 transformed/report/chunk-0-part0.json | python3 -c "
import json,sys
e = json.loads(sys.stdin.readline())
for k,v in e.get('attributes', {}).items():
    if isinstance(v, dict) and 'typeName' in v:
        assert 'uniqueAttributes' in v, f'{k} uses attributes instead of uniqueAttributes'
    if isinstance(v, list) and v and isinstance(v[0], dict) and 'typeName' in v[0]:
        for ref in v:
            assert 'uniqueAttributes' in ref, f'{k}[] uses attributes instead of uniqueAttributes'
print('All refs valid')
"
```

### V3. `extraction-method` Field Is Required in the Contract

**What fails:** Heracles checks `extractionMethod == "direct"` (or `"query_history"`) to route
sage/preflight requests to the native app's `/workflows/v1/check` endpoint. Without this field,
`extractionMethod` is empty, Heracles falls through to the legacy `sageTemplate` curl path.

**Why it's easy to miss:** Configmap, auth, and metadata endpoints all work perfectly. Only the
preflight check endpoint fails — and only when triggered through the Atlan UI (via Heracles),
not when hitting `/workflows/v1/check` directly.

**How to verify:**
- Open `contract/app.pkl` and verify the Credential step has an `extraction-method` field.
- After generating, check the workflow config JSON: `extraction-method` must exist as a property with a default value.

### V4. Pre-commit Must Exclude Generated `_input.py`

**What fails:** Pre-commit hooks (debug-statements, ruff, pyright) parse every `.py` file. The
generated `app/contracts/_input.py` may use imports or patterns that fail, causing CI to break.

**How to verify:**
- Check `.pre-commit-config.yaml` — the debug-statements hook must have `exclude: app/contracts/_input\.py`.
- Check `pyproject.toml` — `[tool.ruff]` and `[tool.pyright]` should exclude `app/contracts/_input.py`.

### V5. Credential Config Fields Must Match Legacy Behavior

**What fails:** If host and port are visible in the credential form but the connector has fixed
values (e.g., `app.mode.com:443` for Mode), users can accidentally modify them and break connectivity.

**How to verify:**
- Read the legacy Argo configmap from `marketplace-packages/packages/atlan/connectors/configmaps/<name>.yaml`.
- Every field marked `"hidden": true` in the legacy config must have `isHidden = true` in the contract's `FieldSpec` (requires toolkit 0.2.6+).
- Every field with a fixed default in the legacy config must have `defaultValue` set in the contract.

### V6. Always Use Latest Toolkit Version

**What fails:** Older toolkit versions have known bugs — e.g., pre-0.2.4 generates `"widget": "textBox"`
instead of `"TextInput"` for textarea fields, pre-0.2.9 generates hyphenated Python field names in `_input.py`.

**How to verify:**
```bash
CURRENT=$(grep -oE '@[0-9.]+' contract/PklProject | tr -d '@')
LATEST=$(gh release list --repo atlanhq/app-contract-toolkit --limit 1 --json tagName --jq '.[0].tagName' | tr -d 'v')
echo "Current: $CURRENT, Latest: $LATEST"
```
If they differ, bump, re-resolve, and regenerate.

### V7. Local Workflow Testing Requires `credential_guid` in Payload

**What fails:** The SDK's `_set_state` only calls `client.load()` when `credential_guid` is
present in workflow args. Without it, the SQL client engine is never initialized and activities
fail with `SQL client engine not initialized`.

**How to test locally:**
```bash
# Start workflow with credential_guid (underscore, top-level) + inline credential dict
curl -X POST http://localhost:8000/workflows/v1/start \
  -H "Content-Type: application/json" \
  -d '{
    "credential_guid": "test-local-cred",
    "credential": {"host": "...", "port": "...", "username": "...", "password": "...", "extra": {...}},
    "connection": {"qualifiedName": "default/<connector>/<name>", "name": "<name>"},
    "metadata": {"extraction-method": "direct", ...}
  }'
```
Key rules: `credential_guid` (underscore, not hyphen), `qualifiedName` (camelCase, not snake_case).

## Reference: FieldSpec Properties

| Property | Type | Default | Description |
|---|---|---|---|
| `name` | String | required | Internal field name (JSON key) |
| `displayName` | String | auto from name | UI label |
| `placeholder` | String? | null | Placeholder text |
| `helpText` | String? | null | Tooltip/help |
| `required` | Boolean | true | Required field |
| `sensitive` | Boolean | false | Mask as password |
| `isHidden` | Boolean | false | Hide field in UI (for fixed-value fields like host/port) |
| `fieldType` | FieldType | "text" | text, password, textarea, select, number, checkbox, url, email |
| `options` | Listing? | null | For select — allowed values |
| `optionLabels` | Mapping? | null | Display names for options |
| `defaultValue` | String? | null | Pre-filled value |
| `validationRegex` | String? | null | Regex pattern |
| `width` | Int | 8 | Grid width (8=full, 4=half) |

### FieldType → Widget Mapping

| FieldSpec | Generated Widget |
|---|---|
| `fieldType = "text"` | `input` |
| `sensitive = true` | `password` |
| `fieldType = "number"` | `inputNumber` |
| `fieldType = "select"` (≤4 options) | `radio` |
| `fieldType = "select"` (>4 options) | `select` |
| `fieldType = "checkbox"` | `boolean` |
| `fieldType = "textarea"` | `TextInput` |

## Reference: Custom DAG Nodes

**Default pipeline (no config needed):** `extract → publish`

**Adding a node between extract and publish:**
```pkl
extraNodes {
  ["lineage"] = new DAGNode {
    displayName = "Extract Lineage"
    workflowType = "LineageExtractionWorkflow"
    taskQueue = "atlan-<name>-{deployment_name}"
    args {
      ["credential"] = "{{credential}}"
      ["connection"] = "{{connection}}"
    }
    dependsOn { "extract" }
  }
  ["publish"] = new PublishNode {
    upstream = "lineage"
  }
}
```

**DAG value types:**
- `{{param-name}}` — Heracles substitutes from form values
- `$.node.outputs.*` — AE resolves from upstream node outputs
- Static values — passed through
